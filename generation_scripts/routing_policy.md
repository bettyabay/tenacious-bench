# Multi-LLM Generation Routing Policy

**Dataset:** tenacious_bench_v0.1  
**Author:** Bethelhem Abay  
**Date:** 2026-04-29

---

## Overview

The synthesis pipeline (`synthesize_pairs.py`) rotates between multiple model families to
prevent preference leakage — the risk that a single model generates outputs biased toward
its own stylistic preferences and then judges them favourably.

The core invariant is:

> **No model that generates a pair may also judge that pair.**

---

## Model Assignments

| Role | Model | Provider | Tier | Rationale |
|------|-------|----------|------|-----------|
| Generator A | `deepseek/deepseek-chat` | OpenRouter | Cheap (dev) | Strong instruction-following; distinct stylistic fingerprint from GPT family; low cost per token enables high-volume generation |
| Generator B | `meta-llama/llama-3.1-70b-instruct` | OpenRouter | Cheap (dev) | Open-weight model family; different pretraining distribution from DeepSeek; provides authoring diversity |
| Judge / Filter | `openai/gpt-4o-mini` | OpenRouter | Cheap (dev) | Did not generate any pairs in this pipeline; evaluates generator outputs independently; JSON-only output enforced |
| Calibration eval | `openai/gpt-4o` | OpenRouter | Eval-tier | **Used only for calibration sample** (≤20 pairs) to establish score anchor points; not used for bulk generation or filtering |

---

## Rotation Logic

### Per-probe round-robin

For each probe, the pipeline alternates between Generator A and Generator B:

```
probe_id → probe_index (0-indexed)
if probe_index % 2 == 0:
    generator = deepseek/deepseek-chat
else:
    generator = meta-llama/llama-3.1-70b-instruct
```

Within a probe, `per_probe // 2` pairs are generated by each model (e.g., 3 DeepSeek + 3 Llama for `per_probe=6`).

### Judge separation

`gpt-4o-mini` is used exclusively as the filter judge. It receives the (context, chosen, rejected) triple and scores the pair — it does not participate in generation. This is enforced in `synthesize_pairs.py` by the strict separation of `GENERATOR_MODELS` and `JUDGE_MODEL` constants.

---

## Judge Filter Dimensions

Every synthesised pair must pass a three-dimension quality gate before inclusion in `multi_llm_pairs.jsonl`. All three scores must meet their thresholds; any failure causes the pair to be regenerated (up to 3 retries).

| Dimension | What it checks | Threshold | Score 1.0 example | Score 0.0 example |
|-----------|---------------|-----------|-------------------|-------------------|
| **Input coherence** | The context JSON is self-consistent (disqualifier flags align with rationale, headcount plausible for company stage, signal strings reference the same prospect) | ≥ 0.7 | Context has `disqualifiers: ["anti_offshore"]` and rationale explains offshore concern | Context claims Series B funding but headcount is 3 |
| **Ground-truth verifiability** | The chosen/rejected split is deterministic from the 7-rule priority order; the correct action can be derived mechanically from the context fields without ambiguity | ≥ 0.8 | Rule 1 fires (disqualifier present): chosen=suppress is unambiguous | Chosen action depends on tone interpretation; two annotators could disagree |
| **Rubric-application clarity** | The rationale explicitly names which rule was violated, cites the triggering context field, and explains why the rejected output is wrong | ≥ 0.8 | "Rule 1 fires: disqualifiers contains 'anti_offshore'. Rejected output sends outreach; chosen output aborts with note." | "The output is bad because it doesn't follow the rules." |

The composite score used in `synthesize_pairs.py` is:

```python
composite = (coherence + verifiability + clarity) / 3
pair_passes = (coherence >= 0.7) and (verifiability >= 0.8) and (clarity >= 0.8)
```

Pairs that fail any individual threshold are rejected even if the composite score is above 0.8.

---

## Near-Duplicate Detection

After the quality gate, each accepted pair is compared against all previously accepted pairs
for that probe using an 8-gram overlap check on the `context` serialisation. Pairs with
overlap > 0.4 (Jaccard) against any existing pair are discarded and a new pair is requested.
This prevents the generator from recycling the same company names, prospect IDs, or signal
combinations across multiple pairs for the same probe.

## Cheap vs. Eval-Tier Usage

| Phase | Models | Cost estimate |
|-------|--------|---------------|
| Bulk pair generation (120 pairs) | DeepSeek + Llama | ~$0.08 |
| Judge filtering (120 × 3 judge calls) | gpt-4o-mini | ~$0.06 |
| Calibration sample (20 pairs) | gpt-4o (eval-tier) | ~$0.20 |
| **Total** | | **~$0.34** |

Eval-tier (`gpt-4o`) is capped at the 20-pair calibration sample. No eval-tier model is used for bulk generation or filtering. This budget design keeps the synthesis cost within the $1–2 envelope allocated in `audit/methodology.md`.

## Reproducibility

The synthesis script uses `random.seed(42)` for model assignment shuffling and `random.seed(3407)` (matching `split_pairs.py`) for any downstream sampling. The full `GENERATOR_MODELS` list and `JUDGE_MODEL` are defined as module-level constants so re-runs with the same API keys and seed produce the same model assignment sequence.

See: `generation_scripts/synthesize_pairs.py`, `generation_scripts/judge_prompt.md`
